SMS 开发路线图 v2.0
=====================

定位修正（2024）
---------------
经过验证，SMS 不是"日常编码工具"——直接对话生成代码比 SMS 快十倍。
SMS 的真正价值在于管理模块的演化，以及给 AI 提供它现在没有的工具。

SMS 双核架构：
  ├── 模块工厂：管理模块演化（Contract匹配、演化追踪、版本管理）
  └── 传统编译器：给 AI 装上"自己验证自己"的能力


Phase 1: 基础框架 ✅ 已完成
---------------------------
  [✓] 知识图谱 (core/)
  [✓] 装配引擎 (assembly/)
  [✓] 标准模块规范
  [✓] 模块注册表 (registry/)
  [✓] 概念注册表 (concept/)
  [✓] 状态流转强制
  [✓] Evidence 行为验证
  [✓] Assembly 自动推导依赖
  [✓] Contract 自动匹配


Phase 2: 双核融合 [当前阶段]
-----------------------------
目标：模块工厂 + 编译器接通，AI 对话生成模块 → 编译器验证

模块工厂侧（砍掉多余，聚焦核心）：
  [ ] 砍掉 Importer（通用Python导入 — 不是SMS该干的）
  [ ] 砍掉 IRCompiler 行为模板（代码生成是编译器的事）
  [ ] 砍掉 Builder 代码生成（交给编译器）
  [ ] 聚焦 Contract 匹配验证
  [ ] 聚焦模块演化追踪（改一个模块，标记受影响的其他模块）
  [ ] 聚焦版本管理（同一模块多版本，产品用哪个版本）

编译器侧（激活 _archive 里的 4500 行编译器）：
  [ ] 分析现有编译器架构
  [ ] 定义编译器与模块工厂的接口
  [ ] 编译器接收 Standard Module，输出编译结果
  [ ] 不只是语法检查，是 Contract 级别的类型验证
  [ ] AI 生成代码 → 编译器检查 → 当场发现问题


Phase 3: 演化验证
-----------------
目标：证明 SMS 的真正价值 — 系统演化时不崩

  [ ] 多版本模块管理
  [ ] 模块替换：A→B，自动验证 Contract 兼容
  [ ] 影响分析：改一个模块，图谱标记所有受影响的产品
  [ ] 回滚：决策变了，自动回到之前的模块组合


Phase 4: 扩展到其他领域
-----------------------
  [ ] 从 Agent 领域扩展到编译器组件
  [ ] 从 Agent 领域扩展到后端服务
  [ ] 多语言编译器后端


核心原则（修正后）
------------------
1. SMS 不是日常编码工具，是演化管理基础设施
2. 模块工厂只管模块间关系，不管代码怎么写
3. 编译器给 AI 用，让 AI 能自己验证自己生成的代码
4. 直接对话写脚本用 Termux，管理多模块系统用 SMS
