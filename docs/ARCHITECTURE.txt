SMS 系统架构 v2.0
==================

双核架构
--------
SMS 由两个核心组成：

第一核：模块工厂 (Module Factory)
  职责：管理模块的演化
  - 知识图谱：记录 Problem → Decision → Module → Product 的完整决策轨迹
  - Contract 匹配：A 的输出和 B 的输入自动验证兼容性
  - 演化追踪：改一个模块，标记所有受影响的产品
  - 版本管理：同一模块多版本共存，产品绑定具体版本

第二核：传统编译器 (Compiler)
  职责：给 AI 装上"自己验证自己"的能力
  - AI 生成代码 → 编译器检查 → 类型/接口/Contract 验证
  - 不只是 Python 语法检查，是 Contract 级别的语义验证
  - 多后端支持：Python / Rust / JS / ...

两个核的关系：
  AI 对话 → 模块工厂（管理模块关系）→ 编译器（验证正确性）→ 可运行代码

SMS 不做什么
-----------
  - 不生成代码（那是 AI 的事）
  - 不写业务逻辑（那是 AI 和人的事）
  - 不替代 Termux 或 Python（日常写脚本直接用它们）

SMS 做什么
---------
  - 管理模块间的关系（谁依赖谁、接口对不对）
  - 保存决策轨迹（为什么选这个模块而不是那个）
  - 验证演化安全（换了模块会不会崩）
  - 给 AI 编译器（AI 现在没有的能力）
